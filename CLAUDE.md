# Frontload

Local-first context and cost gateway for AI coding agents. Frontload indexes a repository, builds task dossiers, provides budgeted reads and summarized command output, and exposes those capabilities through a CLI and MCP server.

## Build

```bash
pnpm build          # TypeScript compile
pnpm test           # unit tests
pnpm e2e            # CLI/MCP e2e test
pnpm proof          # full proof workflow
```

Use `node dist/src/cli/index.js ...` when testing the built CLI directly.

## Testing

- Keep tests close to behavior. Prefer CLI, installer, MCP, and command-summary outcomes over implementation details.
- Add focused unit coverage for installer/config merge behavior, path resolution, plugin validation, and budget calculations.
- Use e2e coverage when changing MCP tool registration, CLI command wiring, or transcript/proof behavior.
- Do not leave generated proof/log timestamp churn in the diff unless the fixture output intentionally changed.
- When validating plugin packaging, run `node dist/src/cli/index.js validate-plugins --repo .`.

## Architecture

- `src/cli/` - Commander CLI entrypoint and command wiring.
- `src/install/` - project init, editor MCP config adapters, skill installation, and global binary checks.
- `src/mcp/` - stdio MCP server and tool registration.
- `src/indexer/`, `src/dossier/`, `src/commands/`, `src/diff/`, `src/budget/` - local context, summarization, and budget features.
- `src/gate/` and `hooks/` - pre-tool-use policy gate.
- `plugins/` - bundled Codex and Claude plugin assets. Keep plugin configs portable; do not reintroduce path-sensitive launcher assumptions.
- `skills/` - shared skill assets. Host-specific plugin skills live under `plugins/<host>/skills/`.

## Product Rules

- Frontload is local-first. Do not add runtime calls that send source code, logs, or summaries to external services.
- MCP stdio servers are launched by the editor/client from config. `frontload init` should configure clients, not start a long-running MCP process.
- Generated MCP config should prefer portable commands, usually the bare `frontload` binary with explicit args.
- Installer writes must be merge-aware and must not clobber unrelated user config.
- If a setup path depends on `frontload` being on `PATH`, verify that condition and provide a manual fallback when it fails.

## Documentation Updates

Any user-visible CLI, init, MCP, plugin, or setup behavior change is not done until docs are checked.

1. Grep `README.md` and `docs/` for the changed feature or command.
2. Update setup docs, troubleshooting, and plugin READMEs when behavior changes.
3. Keep examples executable and aligned with `package.json` scripts.
4. If proof files or transcripts are regenerated by tests, restore timestamp-only churn before committing.

Skip this only for pure internal refactors with no user-visible behavior change.

## Git Workflow

- Keep commits focused and scoped to the requested change.
- Before commit, run `git diff --check` and the narrowest relevant verification.
- If CI, push, or PR checks fail, inspect the failing logs before retrying.
- Never force-push or rewrite history unless the user explicitly asks.
